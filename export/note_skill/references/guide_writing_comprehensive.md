## 記事執筆の段階管理

### 原則
- 各段階（アイデア、執筆中、公開準備完了）において、**1つの記事につき1つのファイルのみを保持する**
- 記事が次の段階に進む際は、古い段階のファイルを削除し、新しい段階にファイルを移動する

### ワークフロー
1. **アイデア段階** → `/02_下書き/01_アイデア/` に記事テーマを記載
2. **執筆開始** → `/02_下書き/02_執筆中/` にファイルを作成（アイデアからは削除）
3. **公開準備** → `/02_下書き/03_公開準備完了/` に移動（執筆中からは削除）

## noteで人気記事を書くための体系的ガイド

### 📊 人気記事の法則

#### 1. タイトルの黄金則
**数字 + 具体的な成果 + ターゲット明確化**

効果的な要素：
- 具体的な数字（期間、金額、倍率など）
- 明確な成果やベネフィット
- 想定読者が自分事と感じる表現

避けるべきパターン：
- 抽象的で曖昧な表現
- 読者メリットが不明確
- ターゲットが広すぎる

#### 2. 冒頭で心を掴む（最重要）
**最初の3行で読者の共感を得る**

効果的な冒頭パターン：
- 読者の悩みや課題の言語化
- 共感を呼ぶ体験談
- 意外性のある事実や数字
- 問いかけによる巻き込み

テクニック：
- 引用ブロック（>）で視覚的に強調
- 短文で畳みかける
- 感情に訴える表現

#### 3. 実績・権威性の提示
**説得力を高める具体的な根拠**

提示すべき要素：
- 定量的な実績（件数、期間、金額）
- 具体的な使用環境や条件
- 客観的な評価や認定
- 実体験に基づく期間や頻度

### 🎨 noteの視覚的工夫

#### 1. コードブロックの活用
**本来の用途以外での視認性向上**

スペック表示に活用：
```
価格: 84,980円（税込）
ディスプレイ: Sony製0.55インチMicro-OLED
リフレッシュレート: 最大120Hz
視野角: 57度
```

手順の表示に活用：
```

2. 自動的に外部ディスプレイとして認識
3. システム設定でディスプレイ配置を調整
```

#### 2. 引用ブロックでメリハリ
**重要な情報や対話形式で使用**

**❗ 重要：引用ブロック内は装飾なしのプレーンテキストのみ**

FAQ形式：
```
> Q: 画面が認識されない
> A: USB-Cポートを変更してみてください。
```

#### 3. URL配置の戦略
**カード表示を活用した信頼性向上**

```

https://jp.shop.xreal.com/products/xreal-one-pro

参考レビュー記事：
https://www.moguravr.com/xreal-one-pro-review/
```

- URLは必ず独立した行に配置
- 前に説明文を付けて文脈を明確化
- noteが自動でカード化し、視覚的に訴求

### 📈 エンゲージメント向上テクニック

#### 1. 構成の最適化

**基本構造（2500-4000文字）**
1. フック（200字）- 共感できる課題提示
2. 問題提起（300字）- なぜ重要か
3. 解決策（1500字）- 具体的な方法
4. 実践結果（800字）- Before/After

#### 2. 読みやすさの工夫

- **見出しは2階層まで**（##と###のみ使用）
- **段落は3-4行で改行**（モバイル対応）
- **重要語句は太字**で強調
- **箇条書きは「・」**を使用（noteでの表示最適化）

#### 3. 信頼性を高める要素

- **具体的な数値データ**
  - 「体感」ではなく「30%短縮」
  - 「たくさん」ではなく「230件」

- **デメリットも正直に記載**
  - 「視野角の端は首を動かす必要がある」
  - 「目頭部分の発熱」
  - 「純正ケースの使い勝手が悪い」

- **実体験に基づく詳細**
  - 「毎日8時間使用」
  - 「3ヶ月間の検証」
  - 使用環境の具体的記載

## 🔥 ストーリー型記事の書き方（最重要）

### なぜストーリー型が必要か

**ストーリー型が効果的な理由：**
- 読者は感情的なつながりを求めている
- 失敗や葛藤を含む話は信頼性が高い
- 変化や成長の過程に共感が生まれる
- 完璧な成功談より学びが多い

### ストーリー型記事の3つの型

#### 1. 体験談型
個人的な体験を軸に展開。感情の動きが重要。

**構成例：**
```
問題発生 → 試行錯誤 → 失敗 → 気づき → 解決 → 学び
```

#### 2. 失敗からの逆転型
最初のつまずきから始めることで読者を引き込む。

**導入例：**
```
「正直に言う。XREAL One Pro、最初の3日間は後悔しかなかった。」
```

#### 3. ビフォーアフター型
過去の自分と現在の自分を対比させる。

**例：**
```
「3ヶ月前まで、僕はMacBookの13インチ画面でUnity開発してた。」
→ 「今では32:9の画面で...」
```

### 人間味を出す文体テクニック

#### 1. 口語表現の活用
- ❌ 「効率が向上しました」
- ⭕ 「マジで効率上がった」

#### 2. 感情表現を入れる
- 「震えた」「泣いた」「ビビった」
- 「〜かも」「〜な気がする」（断定を避ける）

#### 3. 具体的な数字より体感を優先する場面
- 「Command+Tabを100回は押してた気がする」
- 「8万円、ドブに捨てたかも...」

#### 4. 読者への話しかけ
- 「騙されたと思って試してみて」
- 「もし3ヶ月前の自分に会えるなら」

### ストーリー記事の黄金構成

```
1. フック（衝撃的な一文）
   ↓
2. 過去の状況（共感を得る）
   ↓
3. きっかけ（変化の始まり）
   ↓
4. 失敗・葛藤（リアリティ）
   ↓
5. 転機（発見・気づき）
   ↓
6. 成功体験（具体的な効果）
   ↓
7. デメリットも正直に（信頼性）
   ↓
8. 読者へのメッセージ（行動喚起）
```

**❗ 重要：追伸は書かない**
- 追伸を書くと記事が冗長になる
- まとめで完結させるのが望ましい
- CTAの後に追伸があると離脱率が上がる

### 避けるべき「教科書的」表現

#### ❌ 避けるべき例
- 「以下の通りです」
- 「〜することができます」
- 「効果的です」
- 「最適化されています」

#### ⭕ 代わりに使う表現
- 「こんな感じ」
- 「〜できる」
- 「めっちゃいい」
- 「最高になった」

### 実際のリライト事例

#### Before（教科書的）
```
「こんな悩みを抱えているUnity開発者の皆さんに朗報です。」
```

#### After（ストーリー型）
```
「3ヶ月前まで、僕はMacBookの13インチ画面でUnity開発してた。
タブ切り替えだけで1日100回はCommand+Tab押してた気がする。」
```

## 🎨 noteで使える装飾テクニック完全ガイド

### 装飾の基本原則
**noteは過度な装飾を嫌うプラットフォーム**だが、適切な装飾で読みやすさは劇的に向上する。
読者の「視線の流れ」を意識した装飾が重要。

### 1. 章立て・見出しの工夫

#### 【第○章】形式
```
## 【第1章】最初の数日間は調整に苦労した
## 【第2章】世界が変わった瞬間
## 【第3章】散歩しながら使ってみた
```
- ストーリーの流れを明確化
- 読者が現在地を把握しやすい
- スクロールしても迷わない

#### 見出しレベルの使い分け
```
# メインタイトル（1つだけ）
## 大見出し（章タイトル）
### 小見出し（節タイトル）
```
noteでは3階層までが推奨。それ以上は読みにくくなる。

### 2. 強調表現のバリエーション

#### 太字の効果的な使い方
```
**21:9のウルトラワイドモニター** → 製品スペック
**自分だけの世界** → 感情的な強調
**32:9は別次元** → インパクトのある表現
```
1段落に1〜2個まで。多用すると効果が薄れる。

#### 数字の見せ方
```
❌ 二十万円
⭕ 20万円
⭕ **20万円**（さらに強調）
```
漢数字より算用数字の方が視認性が高い。

### 3. 引用ブロック（>）の活用法

**❗ 重要：引用ブロック内は装飾なしのプレーンテキストのみ**

#### 内心の声・つぶやき
```
> ...あれ？
> これ、移動時間が全部学習時間になるじゃん
```

#### 重要なメッセージ
```
> 迷ってないで、さっさと買え
```

#### 悩みの可視化
```
> 32:9のウルトラワイドモニターに買い替えるか...
```

引用ブロックは「読者の目を止める」効果がある。
ただし、引用ブロック内に太字（**）やその他の装飾を使用すると、noteでバグが発生するため使用しない。

### 4. リスト表現のテクニック

#### 番号付きリスト（特殊文字版）
```
❶ リクライニング開発の革命
❷ 移動時間が開発時間になった
❸ 集中力が異常に上がった
```
普通の1,2,3より視覚的インパクトが強い。

#### 箇条書きの階層化
```
・親項目
  ・子項目1
  ・子項目2
    ・孫項目
```
インデントで情報の階層を表現。

#### チェックマーク活用
```
✓ 持ち運べる（87g）
✓ 電源いらない
✓ 設置スペースいらない
```
ポジティブな要素に使用。

#### 三角マーク活用
```
▼ 発熱する
▼ 端っこは見づらい
▼ 純正ケースがクソ
```
ネガティブな要素・注意点に使用。

### 5. コードブロックの創造的活用

#### 図解・ビジュアル表現
```
[ Visual Studio Code ][ Unity Editor ][ ChatGPT ]
←――――――― 32:9の広大な画面 ―――――――→
```

#### 比較表の代替
```
【32:9ウルトラワイドモニター】
・Samsung Odyssey G9: 約21万円
・LG 49WL95C-W: 初値15万円

【XREAL One Pro】
価格: 84,980円
差額: 6万円〜12万円
```

#### フロー図の表現
```
月の移動時間：合計20時間
　　↓
フル効率の開発時間に変換
```

#### 重要設定の強調
```
距離設定: 10メートル ← これが最重要！
解像度: 3840×1080
リフレッシュレート: 60Hz
```

### 6. 記号・絵文字の効果的な使い方

#### 使うべき記号
```
✓ チェックマーク（ポジティブ）
▼ 下向き三角（ネガティブ）
❶❷❸ 数字（順序・重要度）
← → 矢印（注目・フロー）
【】隅付き括弧（カテゴリ分け）
```

#### 絵文字は原則使わない
noteでは絵文字は「素人っぽく」見える。
記事全体で絵文字は避け、記号（✓▼❶など）を使用する。

### 7. 改行・空白の使い方

#### 段落間の空白
```
文章ブロック1
（1行空白）
文章ブロック2
```
詰まりすぎると読みにくい。

#### セクション間の区切り
```
---
```
大きな話題転換時に使用。

### 8. URLの配置戦略

#### 独立行配置
```
**公式ショップはこちら：**
https://jp.shop.xreal.com/products/xreal-one-pro
```
- 説明文を前に配置
- URLは独立した行に
- noteが自動でカード化

### 9. 文章の視覚的リズム

#### 短文・長文の組み合わせ
```
短い。
もう少し長い文章を書く。
そして、さらに詳しい説明を含む長めの文章で、読者に詳細な情報を提供する。
また短く。
```
単調にならないようリズムを作る。

### 10. 実践的な装飾の組み合わせ例

#### Before（装飾なし）
```

## 📝 noteプラットフォーム向け引用ブロック最適化ガイド

### 基本方針：引用ブロック優先使用

**noteでは引用ブロック（>）がコードブロック（```）より読みやすい**

#### 使い分けの原則

**✅ 引用ブロック（>）を使用すべき場面：**
- 例文・サンプル
- 比較表示（Before/After）
- ユーザーの声・体験談
- 統計データ・数値
- 手順の説明（技術的でない場合）
- 強調したい重要情報
- フロー図・プロセス説明

**✅ コードブロック（```）を使用すべき場面：**
- 実際のプログラムコード
- コマンドライン操作
- 設定ファイルの内容
- 技術的な手順（システム操作）

### 実践的な使い分け例

#### ❌ 避けるべき使用法
```
// 体験談をコードブロックで囲む（読みにくい）
```
「今日は音声入力を試してみました。
思ったより精度が高くて驚きました。」
```
```

#### ⭕ 推奨する使用法
```
// 体験談は引用ブロックで
> 「今日は音声入力を試してみました。
> 思ったより精度が高くて驚きました。」
```

### 引用ブロック内での注意点

#### 0. 装飾なしのプレーンテキストのみ使用（最重要）
**❗ 引用ブロック内は必ずプレーンテキストのみ**

**⭕ 正しい使用法：**
```
> Chrome：100%（バージョン50以降）
> Firefox：100%（バージョン44以降）
> Safari：100%（iOS 11以降）
```

**❌ 避けるべき使用法：**
```
> **Chrome**：100%（バージョン50以降）
> **Firefox**：100%（バージョン44以降）
> **Safari**：100%（iOS 11以降）
```
→ 引用ブロック内に太字（**）などの装飾を使用すると、noteでバグが発生する

**理由：**
- 引用ブロック内の装飾はnoteで正しく認識されない
- コピペ時にレイアウトが崩れる
- 視認性向上は引用ブロック自体で十分

#### 1. 数字箇条書きの回避
**❌ 問題のあるパターン：**
```
> 1. 手順その1
> 2. 手順その2
> 3. 手順その3
```
→ コピペ時に箇条書きと認識されてバグる

**⭕ 解決策：**
```
> ❶ 手順その1
> ❷ 手順その2
> ❸ 手順その3
```
または
```
> ❶｜手順その1
> ❷｜手順その2
> ❸｜手順その3
```

#### 2. 空の引用ブロック行の削除
**❌ 問題のあるパターン：**
```
> 文章1
>
> 文章2
```
→ 単独の「>」がコピペ時に空白文字になる

**⭕ 解決策：**
```
> 文章1
>
> 文章2
```
→ 空の引用ブロック行（>のみ）を削除

#### 3. 囲み数字・記号の活用
**推奨記号：**
- **❶❷❸❹❺** ：囲み数字
- **｜** ：区切り文字
- **→** ：矢印（フローの表現）
- **▼** ：注意点・ネガティブ要素
- **✓** ：チェックマーク・ポジティブ要素

### コードブロック使用時の注意点

#### 技術的内容のみに限定
```csharp
// 実際のコード例
public class PlayerController : MonoBehaviour
{
    // 実装内容
}
```

```

❷ 音声認識が起動
❸ 指示を話す
```

### 水平線（---）の使用制限

**noteでは水平線が表示を阻害する可能性があるため、原則として使用しない**

#### ❌ 避けるべき使用
```

---

内容...

---
```

#### ⭕ 代替案
```

内容...

```

### コピペ最適化チェックリスト

記事完成時の確認項目：

- [ ] 引用ブロック内に装飾（太字**、斜体*など）がないか（最重要）
- [ ] 引用ブロック内に数字箇条書き（1. 2. 3.）がないか
- [ ] 空の引用ブロック行（>のみ）がないか
- [ ] 水平線（---）を削除したか
- [ ] URLは独立行に配置したか
- [ ] 技術的でない内容はコードブロックではなく引用ブロックを使用したか
- [ ] 囲み数字（❶❷❸）や記号（｜→）を適切に使用したか
- [ ] 追伸を書いていないか（CTAで終了）

### noteプラットフォーム特有の制約

1. **引用ブロック内の装飾認識（最重要）**
   - 引用ブロック内の太字（**）や斜体（*）は正しく認識されない
   - noteでバグが発生し、表示が崩れる
   - 引用ブロック内は必ずプレーンテキストのみ使用

2. **引用ブロック内の自動箇条書き認識**
   - 「1. 2. 3.」形式は自動的に箇条書きとして認識される
   - コピペ時にレイアウトが崩れる原因となる

3. **コードブロックの視認性**
   - 黒背景で目に負担がかかる
   - 長文には不適切

4. **カード表示機能**
   - URLは独立行配置で自動カード化
   - 視覚的な訴求力が向上

### 修正コマンド例

空の引用ブロック行を一括削除：
```bash
sed -i '' 's/^>$//' filename.md
```

この知識により、noteでの読みやすさと機能性を両立した記事作成が可能になる。

4) 実行（Do）
- 計画の手順に沿って実装

5) 検証（Check）
- チェックリストで品質/整合性を確認
- 失敗時は計画へロールバック（手順修正）

6) 反映（Act）
- ナレッジ更新が必要なら「変更理由/差分/根拠」を併記して該当ファイルに追記
