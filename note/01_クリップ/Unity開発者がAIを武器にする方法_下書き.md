## Pure C#設計がAI活用の鍵

### なぜPure C#か

UnityのMonoBehaviourに依存したコードは：

- テストしにくい
- AIが理解しにくい（Unity固有のコンテキストが必要）
- 再利用性が低い

**Pure C#で書くと：**

- AIが純粋なC#として理解できる
- ユニットテストが書ける
- ロジックとプレゼンテーションが分離される

### 実践例：ゲームロジックの分離

```csharp
// Bad: MonoBehaviourに密結合
public class PlayerController : MonoBehaviour
{
    public int hp = 100;
    public void TakeDamage(int damage)
    {
        hp -= damage;
        if (hp <= 0) Die();
    }
}

// Good: Pure C#でロジックを分離
public class PlayerHealth
{
    public int Current { get; private set; }
    public int Max { get; }
    public bool IsDead => Current <= 0;

    public PlayerHealth(int max)
    {
        Max = max;
        Current = max;
    }

    public void TakeDamage(int damage)
    {
        Current = Math.Max(0, Current - damage);
    }
}
```

Pure C#ならAIに「テストコード書いて」と頼むだけでテストが生成される。

## 陥りやすい落とし穴

### 1. AIに丸投げしてしまう

**問題:** 設計思想なしにコードを生成すると、統一感のないコードベースになる。

**対策:** 設計ルールを先に決め、AIにはそのルール内で動いてもらう。

### 2. AIの出力を検証しない

**問題:** AIは自信満々に間違ったコードを書く。

**対策:** 必ずレビュー。特にエッジケース、null参照、非同期処理。

### 3. 全部AIでやろうとする

**問題:** エディタ操作、アセット配置などはAIに向かない。

**対策:** コード生成に特化させ、それ以外は人間がやる。
