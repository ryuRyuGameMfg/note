## 解決の鍵は「Pure C#」だった

### MonoBehaviourが多すぎる問題

あるとき、ふと気づいたんです。

> 「なんでこんなにアタッチ作業が多いんだ？」

答えはシンプルでした。MonoBehaviourが多すぎるんです。

Unityは「コンポーネント」という設計思想で作られてます。
MonoBehaviourを継承したスクリプトは、ゲームオブジェクトにアタッチできるんですよね。

これ、プログラマーじゃなくても触れるように設計されてるんです。
デザイナーやプランナーでも数値を調整できます。
Unityが広まった理由のひとつだと思います。

**でも、AI時代にはこれが逆に足を引っ張るんですよ。**

AIは10個のMonoBehaviourを書いてきます。
だから僕は10個のアタッチ先を考えなきゃいけない。
10個のインスペクター設定を把握しなきゃいけない。

じゃあ、どうすればいいんでしょうか？

### Pure C#って何？

**MonoBehaviourを減らせばいいんです。**

ロジックは全部Pure C#で書きます。
人間が設定しなきゃいけない部分だけ、MonoBehaviourに集めます。

これが「Pure C#設計」の原点でした。

Pure C#っていうのは、MonoBehaviourを継承しない普通のC#クラスのことです。

```csharp
// MonoBehaviour（Unityにアタッチできる）
public class PlayerController : MonoBehaviour { }

// Pure C#（普通のクラス）
public class PlayerLogic { }
```

Pure C#はUnityのStart()やUpdate()に依存しません。

だからUnityエディタを開かなくてもテストできますし、AIにとっても「普通のC#」だから理解しやすいんです。

> AIが得意な領域はPure C#に任せて、人間がやるべき設定だけMonoBehaviourに残す。

この分業を徹底したら、どうなったと思いますか？

## どうやって実現する？

考え方はシンプルです。

> 「人間が触る場所」と「AIが触る場所」を完全に分ける。

### 攻撃処理で比べてみる

従来のやり方だと、こうなりがちですよね。

```csharp
// 従来：全部MonoBehaviour
public class PlayerAttack : MonoBehaviour
{
    [SerializeField] private int damage = 10;

    void Update()
    {
        if (Input.GetKeyDown(KeyCode.Space))
            Attack();
    }

    private void Attack() { /* 攻撃ロジック */ }
}
```

攻撃ロジックを変えるたびに、Unityエディタで動作確認が必要になります。

Pure C#設計だとこうなります。

```csharp
// Pure C#：ロジックだけ（AIが書く）
public class AttackLogic
{
    public void Execute() { /* 攻撃処理 */ }
}

// MonoBehaviour：Unityとの接点だけ（人間が管理）
public class PlayerManager : MonoBehaviour
{
    private AttackLogic attackLogic = new();

    void Update()
    {
        if (Input.GetKeyDown(KeyCode.Space))
            attackLogic.Execute();
    }
}
```

違いがわかりますか？

**AttackLogicはUnityに依存してないんです。**

だからAIに「攻撃処理を変えて」と言えば、Unityを知らなくても修正できます。
テストもエディタなしで書けます。

人間が見るのはPlayerManagerだけ。
ここにインスペクター設定が集約されてるんです。

### 人間とAI、どう分ける？

**人間が担当するのは上流だけです。**

MonoBehaviour、インスペクター設定、ゲームオブジェクトへのアタッチ。
数を絞って、ここだけ管理します。

**AIが担当するのは中流〜下流。**

Pure C#のビジネスロジック、通信処理、データ処理、テスト。
ここは全部任せます。

> 「必要なものを外から渡す」、これだけなんです。

### おすすめのフォルダ構成

```
Assets/Scripts/
├── MonoBehaviours/     ← 人間が管理（少数）
│   ├── GameManager.cs
│   ├── UIManager.cs
│   └── DebugManager.cs
│
├── PureCS/             ← AIが管理（大量）
│   ├── Logic/
│   ├── Services/
│   └── Utils/
│
└── ScriptableObjects/  ← 設定データ
```

ポイントは **MonoBehavioursフォルダを分離すること** です。

こうすると「MonoBehaviourは少なく保つ」という意識が自然と生まれます。
